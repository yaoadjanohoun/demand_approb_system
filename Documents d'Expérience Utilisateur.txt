Documents d'Expérience Utilisateur (UX) — Système de Demandes et d'Approbation
Ce document définit les principes ergonomiques, les parcours utilisateurs et les standards d'accessibilité pour garantir l'adoption du système par tous les employés, indépendamment de leur rôle ou de leurs capacités. 

1. Charte d'Ergonomie et Principes Directeurs
L'objectif n'est pas de réduire artificiellement le nombre de clics, mais de rendre chaque clic significatif et rassurant.  Les recherches de 2026 confirment que la "règle des 3 clics" est un mythe : les utilisateurs acceptent des parcours plus longs si la progression est claire et logique. 

1.1. Principe de "Progression Visible" (Remplacement de la règle des 3 clics)
Au lieu de compter les clics, nous mesurons la clarté du parcours. 

Fil d'Ariane (Breadcrumbs) : Obligatoire sur tous les écrans. Ex: Accueil > Mes Demandes > Note de Frais > Validation.
Barre de Progression : Pour les formulaires complexes (plus de 5 champs), afficher une barre de progression segmentée (ex: "Étape 2 sur 3 : Pièces jointes").
Feedback Immédiat : Chaque action (clic, validation, erreur) doit avoir une réponse visuelle en moins de 100ms (changement de couleur, spinner, message de succès).

2. Parcours Utilisateurs et Wireframes (Maquettes Fonctionnelles)
Cette section décrit la structure des écrans clés pour les deux rôles principaux.

2.1. Parcours Demandeur : "Soumission en moins de 2 minutes"
L'objectif est de réduire la friction à la saisie.

Écran A : Tableau de Bord Demandeur

En-tête : Bouton d'action principal bien visible : [+ Nouvelle Demande].
Corps : Liste des "Demandes en cours" avec code couleur statut (Vert = Approuvé, Orange = En attente, Rouge = Rejeté).
Accès Rapide : Section "Modèles fréquents" (ex: si l'utilisateur fait souvent des notes de frais, le formulaire pré-rempli apparaît ici).
Écran B : Formulaire Dynamique (Cœur du système)

Structure : Les champs s'affichent progressivement selon le form_schema (JSON).
Validation Inline : Les erreurs (ex: "Montant invalide") apparaissent sous le champ concerné dès la perte de focus, pas seulement à la soumission.
Bouton d'Action : Le bouton "Soumettre" est désactivé (grisé) tant que tous les champs obligatoires ne sont pas valides.


2.2. Parcours Approbateur : "Décision en 1 clic"
L'objectif est de permettre une validation rapide, même sur mobile.

Écran C : File d'Attente d'Approbation

Vue Liste : Tableau synthétique avec colonnes : Date, Demandeur, Type, Montant, Urgence.
Tri Intelligent : Par défaut, trier par "Date la plus ancienne" (pour éviter les blocages) et mettre en surbrillance les demandes dépassant le SLA (48h).
Écran D : Détail de Validation (Mobile First)

Haut de page : Résumé exécutif (Qui, Quoi, Combien).
Pièces Jointes : Aperçu direct des factures/documents sans téléchargement (lightbox).
Zone de Décision Sticky : Les boutons [APPROUVER] (Vert) et [REFUSER] (Rouge) restent toujours visibles à l'écran, même lors du défilement (sur mobile).
Commentaire : Champ texte optionnel qui n'apparaît en grand que si "Refuser" est cliqué (obligatoire en cas de refus).
3. Stratégies d'Accessibilité (Conformité WCAG 2.2 Niveau AA)
Conformément aux directives de 2026, le système doit être utilisable par tous, y compris les personnes en situation de handicap temporaire ou permanent.

3.1. Navigation au Clavier et Focus
Focus Visible : Tout élément interactif (bouton, lien, champ) doit avoir un contour de focus clair et contrasté lorsqu'il est sélectionné au clavier (Tab). Critère WCAG 2.2 : Focus Not Obscured.
Ordre de Tabulation : L'ordre logique doit suivre le flux visuel (Haut vers Bas, Gauche vers Droite).
Raccourcis : Possibilité de valider une demande avec la touche "Entrée" et de refuser avec "Échap" (après confirmation). 
3.2. Contraste et Visibilité
Contraste des Couleurs : Ratio minimum de 4.5:1 pour le texte normal et 3:1 pour les gros titres. Ne jamais utiliser uniquement la couleur pour indiquer un statut (ex: ajouter une icône "Check" pour le vert et "Croix" pour le rouge).
Taille des Cibles : Sur les écrans tactiles (mobiles des commerciaux), les boutons d'action doivent faire au moins 24x24 pixels (nouveau critère WCAG 2.2 Target Size Minimum).
3.3. Compatibilité Lecteurs d'Écran
Balises ARIA : Les zones dynamiques (chargement des formulaires JSON, messages d'erreur inline) doivent utiliser aria-live="polite" pour annoncer les changements aux utilisateurs non-voyants.
Textes Alternatifs : Toutes les icônes et pièces jointes doivent avoir un texte alternatif descriptif.
4. Matrice de Validation UX
Avant le développement, chaque écran sera validé contre ces critères :

Critère	Description	Méthode de Test	Cible
Efficacité	Temps moyen pour soumettre une demande standard.	Test utilisateur chronométré (5 personnes).	< 2 minutes
Satisfaction	Score de facilité d'utilisation (SEQ).	Questionnaire post-tâche (1 à 7).	> 6/7
Accessibilité	Conformité WCAG 2.2 AA.	Audit automatique (axe DevTools) + Test clavier.	100% Critères
Erreur	Taux d'erreur de saisie au premier essai.	Analyse des logs de validation formulaire.	< 5%

5. Recommandations pour l'Implémentation Django
Pour respecter ces principes UX dans le code :

Templates : Utiliser un système de composants Django réutilisables (ex: {% render_field %}) pour garantir que tous les champs ont le même style de focus et d'erreur.
HTMX ou Alpine.js : Pour la réactivité (validation inline, affichage dynamique) sans la complexité d'un framework JS lourd, tout en restant compatible avec l'architecture Django traditionnelle.
Tests d'Interface : Intégrer des tests pytest-selenium pour vérifier automatiquement que les contrastes de couleurs et les attributs ARIA sont présents à chaque déploiement. 
Ce document servira de référence pour les développeurs front-end et les intégrateurs, assurant que la puissance du moteur de règles (backend) est servie par une interface simple et inclusive.
